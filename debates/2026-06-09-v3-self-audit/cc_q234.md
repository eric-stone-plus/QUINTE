## A. Loop-until-dry thresholds — meaningful or cosmetic?

The four thresholds are **meaningful, not cosmetic**, but with one caveat.

**Dry=2 rounds** (PROTOCOL.md:123-124): Two *consecutive* rounds with zero new claims is the convergence criterion for Phase 5. The number 2 is principled — a single dry round could be a stochastic fluke; two consecutive rounds is genuine saturation. It's paired with a second independent condition (dispute count not increasing + evidence repetition >90%), forming a dual-condition termination gate. Both must hold *simultaneously* to trigger escalation. This is a real convergence detector, not decoration.

**Claims ≤ 100 / Refutations ≤ 50 / Loops ≤ 5** (PROTOCOL.md:146): These are in the Governance Layer as **cost circuit breakers**, not convergence criteria. When breached, they trigger "hm approval required to continue" — a blocking gate, not auto-termination. They serve a concrete function: preventing unbounded resource consumption in runaway debates.

**Caveat — the specific numbers (100/50/5) are uncalibrated.** There's no empirical derivation cited. They could be tuned based on observed debate behavior. But they're enforceable via the Workflow engine (`agent` count tracking, phase iteration counters), and the action they trigger (hm synchronous veto gate) is real protocol machinery. So: structurally meaningful, numerically arbitrary.

## B. Cross-model adversarial verification in a DeepSeek-only environment

PROTOCOL.md requires ≥1 of 3 R2 refuters to use a different provider/model (§2 Phase 3:101-103, Invariant §5:198). The current environment has only DeepSeek-v4-pro configured (§4.1:159). This is a **gap between specification and implementation**.

**The most honest answer**: the requirement cannot currently be satisfied. Options, in descending order of fidelity:

| Option | Fidelity to spec | Effort |
|--------|:---:|:---:|
| **1. Provision a second provider** — add an Anthropic API key and use `claude -p --model claude-sonnet-4-6` for one refuter | Full | Low (API key + config) |
| **2. Use a local open-weight model** — run via Ollama/Llama.cpp as the alternative refuter | Full | Medium (setup + GPU) |
| **3. Use different DeepSeek variants** — if DeepSeek offers distinct model versions (e.g., v3 vs r1 vs v4-pro) with genuinely different architectures | Partial | Low |
| **4. Temperature + prompt divergence** — same model, radically different temperature (0 vs 1.0) and adversarial prompt templates | Weak | Trivial |
| **5. Document as a known limitation** — run R2 with same-model refuters but flag the violation in structured log | None (honesty) | None |

Options 1–2 satisfy the invariant. Option 3 depends on whether DeepSeek variants are architecturally distinct enough to count as "different model." Options 4–5 don't satisfy the invariant — they're stopgaps. The protocol already has a mechanism for this: hm's ABORT authority. If cross-model diversity is impossible, hm should ABORT the debate or approve it with a documented invariant violation.

**Specific implementation**: If option 1 is chosen, modify the Phase 3 `parallel()` call so one of the three refutation agents dispatches via `claude -p --model claude-sonnet-4-6` (Bash) instead of `deepseek-v4-pro`. The Workflow script tracks which refuter used which model and includes that in the phase output for hm's review.

## C. KANSA README vs PROTOCOL.md Phase 6 — match analysis

**They are complementary but not identical.** Key divergences:

### What matches:
- Both agree Phase 6 is R3, the KANSA audit layer, operating within QUINTE
- Both agree hm holds final approval authority
- Both agree the audit is a secondary review of prior-round evidence

### What's in PROTOCOL.md but NOT in KANSA README:
1. **Four concrete audit functions**: topic-rotation audit, authorization boundary check, poison detection, gate compliance verification
2. **Output artifact**: structured log written to `~/.hermes/quinte/`
3. **Launch method**: KANSA persona dispatched via `hermes chat -q`
4. **Poison detection heuristics**: single agent >50 claims → anomaly; assertion without evidence → downgrade

### What's in KANSA README but NOT in PROTOCOL.md:
1. **Rotating audit consul matching table** (topic→consul: ledger→omp, contracts→cc, code→cw, protocol→rx) — a whole architectural pattern absent from the spec
2. **Independent registry Agent** for consul matching — not mentioned in PROTOCOL.md
3. **Parallel verdict + merge**: consul drafts verdict alongside hm; consensus→adopted, disagreement→annotated dissent. PROTOCOL.md describes hm as sole approver, not a merge of two parallel verdicts
4. **Design principle**: "rotating second auditor prevents any single agent from accumulating imperial authority" — a governance motivation not stated in the spec

### Verdict: **Partial match with drift.**

PROTOCOL.md Phase 6 describes KANSA as a single persona (`hermes chat -q`) performing four checks with hm final approval. KANSA README describes a richer two-auditor merge architecture (consul + hm, parallel verdicts, merge-and-dissent). These aren't contradictory — the consul could be the mechanism underlying the "KANSA persona" — but the KANSA README introduces details (rotating consul matching table, registry Agent, dual-verdict merge) that PROTOCOL.md doesn't codify. The spec is the canonical definition per its own preamble; the README appears to be an earlier or parallel design document that hasn't been fully absorbed into the v3.0 spec. The consul matching table and merge mechanism should be either incorporated into PROTOCOL.md Phase 6 or marked as superseded.
